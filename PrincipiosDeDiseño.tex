    \subsection{Don't repeat yourself(DRY)}

    Simple. No hay que escribir código duplicado. El código duplicado es propenso a generar errores y es difícil de mantener.
    \subsection{Keep It Simple Stupid(KISS)}
     El diseño de un programa debe ser sencillo. Cuánto más sencillo sea, mejor. Hay que evitar la complejidad como norma general, pero sobre todo la complejidad innecesaria.
     \subsection{SOLID}
    
    \subsubsection{Single Responsibility Principle (SRP)}

    Una clase o módulo debe tener una sola razón para cambiar, porque debe tener una sola responsabilidad. Además esa clase debe ser la única con esa responsabilidad. Es decir, si tenemos que cambiar una clase, que sea por una única razón.
    \subsubsection{Open/Close Principle}
    El código está mejor diseñado si se puede modificar su comportamiento sin cambiar su código fuente. Debería bastar con añadir código, en lugar de modificarlo. No deberíamos necesitar modificar una clase a no ser que sea para corregir errores.
    \subsubsection{Principio de sustitución de Liskov}

   Este principio dice que una clase derivada no debe modificar el comportamiento de la clase base.
   \subsubsection{Interface segregation principle(ISP)}
   Una clase que implementa una interfaz no debe depender de métodos que no utiliza. Esto como norma general implica que nuestras interfaces deben ser sencillas y tener pocos métodos. Es preferible tener varias interfaces pequeñas, a tener una interfaz grande. Así no obligamos a los clientes a depender de métodos que no necesitan implementar.
   \subsubsection{Dependency Inversion Principle(DIP)}
    Las clases de alto nivel, no deben depender de clases de bajo nivel. Ambos deben depender de abstracciones. Además las abstracciones no deben depender de los detalles, si no que los detalles deben depender de las abstracciones. 

    \subsection{Separation of Concerns(SoC)}
    Separa los asuntos,estos son los diferentes aspectos de la funcionalidad de nuestra aplicación.
    \subsection{You ain't gonna need it(YAGNI)}
   No debemos implementar algo si no estamos seguros de necesitarlo. Así ahorramos tiempo y esfuerzo.

   \subsection{Boy Scout Rule}
    
    Como progamadores, debemos hacer como los Boy Scout, que dejan el campo más limpio que cuándo llegaron. El código siempre puede mejorar. Si podemos, debemos refactorizar el código para dejarlo todavía más limpio y simple que antes.
    \subsection{Demeter's Law}
    Una unidad solo debe tener conocimiento limitado de otras unidades, y solo conocer aquellas que están relacionadas. La una unidad solo debe hablar con amigos y nunca con extraños. Además la unidad solo debe hablar con amigos inmediatos.

    \subsection{Hollwood's Principle}
   "No nos llame, nosotros le llamaremos". Este principio está relacionado con el principio de inversión de dependencias de SOLID. Un ejemplo del principio de Hollywood es la inversión de control (IoC), que hace que una clase obtenga las referencias a objetos que necesita para funcionar, a través de una entidad externa.
    

    \section{Principio de Diseño de Software}
    \begin{itemize}
        \item Es una secuencia de pasos donde interviene la creatividad, experiencia y compromiso con la calidad.
        \item Tiene factores decalidad externos tales como las propiedades que pueden ser observadas por los usuarios.
        \item Tiene factores de calidad internos que son buscados por el Ingeniero de Software.
    \end{itemize}


    \section{Principio de Diseño de Software Segun Alan Davis}
    \begin{itemize}
        \item En el proceso deben tomarse enfoques alternativos
        \item Deberá rastrearse hasta el análisis
        \item Se debe reutilizar
        \item Tratar de imitar el dominio del problema
        \item Uniformidad e integración.
        \item Deberá estructurase para admitir cambios.
        \item Debe prever la adaptación a circunstancias inusuales.
        \item No codificar.
        \item Evaluarse en función de calidad mientras está creciendo.
        \item Minimizar errores conceptuales.
    \end{itemize}
    \section{Conceptos de diseño}
    \subsection{Abstracción}
    Manera en que se representa un objeto de la vida real.
    \subsubsection{Tipos de abstracción}
    \begin{itemize}
        \item Procedimental:

        Es una secuencia de instrucciones que tiene una función específica y limitada.

        \item De Datos:

        Colección de datos que describe un objeto de datos
        \item De Control:

        El mecanismo de control de programa sin especificar datos internos.
    \end{itemize}

    \subsection{Refinamiento}
    Se realiza paso a paso, obliga a que el diseñador trabaje sobre la sentencia original, proporcionando cada vez más detalles a medida que suceden los refinamientos.

    \subsection{Modularidad}
    Se divide el software en componentes reutilizables.

    \subsection{Arquitectura de Software}
    Sirve como marco de trabajo desde donde se llevan a cabo actividades de diseño más detallada.
