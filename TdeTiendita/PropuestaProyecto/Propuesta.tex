    \section{Introducción}

        Como proyecto a trabajar en la unidad de aprendizaje de Análisis y Desarrollo Orientado a Objetos
        nos disponemos a crear un Sistema basado en Web (usando sobretodo NodeJS, React y Materialize)
        para poder gestionar un pequeño negocio de abarrotes y papelería.
        
        
        Será capaz de poder guardar los registros de compra de productos basados en una captura de código de barras en
        un base de datos local y poder generar los reportes correspondientes.
        
        
        Se pretende un sistema de fácil uso, con caracteristicas enfocadas a las necesidades particulares de los clientes a los que se les entregará el sistema.
        
        
        Un Sistema de Punto de Venta es la versión computarizada de una caja registradora tradicional o de un cuaderno para llevar las cuentas, que gracias a la tecnología actual, añade muchas funcionalidades más que antes no existían o dependían de sistemas muy complejos, grandes y caros. Ahora su operación y precios son mucho más accesibles.
        
        
        Comúnmente está compuesto por un dispositivo central (computadora, smartphone, tablet) y varios dispositivos de entrada y salida (escáner de códigos, impresora, cajón de dinero, terminal de cobro para tarjeta de crédito o débito). Cada vez son más compactos y poderosos.
        
        
        Con esto en mente propusimos el siguiente proyecto:

        a mano. En nuestro caso, para nuestro proyecto de Análisis y Diseño Orientado a Objetos
        pensamos en aprovechar la oportunidad de crear un proyecto computacional hecho y derecho
        para actualizar el sistema que se ocupa en la tienda de abarrotes de mi familia.
    
    
        
    % ==============================================================
    % =========            PROBLEMATICA                =============
    % ==============================================================
    \section{Problemática}
    
        Debido a la gran cantidad de información que se maneja en el negocio familiar buscamos en usar
        una computadora y una aplicación de contabilidad y venta para que pudieramos usar los beneficios
        que trae la informática.
    
        Han pasado muchos años y la tecnología (sobretodo en estas últimas decadas) crece a niveles
        exponenciales, por lo tanto a pesar de que apenas han pasado unos pocos años desde que instalamos
        el sistema ya se siente muy obsoleto, por el requerir de software como VisualFox 95 o una \Quote{base}
        de datos en MS Access.
        
        Por lo mismo pensamos en actualizar el sistema para colocar otro que cumpla mejor nuestos
        requerimientos, pero nos encontramos en que casi todos eran de costos muy elevados para
        ser usados y finanzados con las ganancias de una pequeña tiendita de la esquina.
        
        Además todos aquellos han sido diseñados para trabajar en miles y miles de terminales, por lo que
        la interfaz gráfica es lo de menos y buscan mucho mejor que su software pueda correr en terminales
        cada vez más baratas y que solo se pueden usar para lo mismo, pero ese no es completamente nuestro
        caso, pues disponemos de una computadora más que decente pero no podemos aprovechar sus beneficios
        si todas las aplicaciones en el mercado estan mucho más enfocadas en que todo sea lo más eficiente
        posible.
        
        Los clientes que se nos presentan tienen un perfil de conocimiento en software medio de entre intermedio y avanzado, son conformados por una familia y empleados ocasionales.
        
        
        Nos enfrentamos a la dificultad de comunicación entre el escáner de datos y el sofware a realizar, la seguridad de la base de datos y gestionamiento de usuarios.
    
    
    % ==============================================================
    % =========          SOLUCIONES EXISTENTES         =============
    % ==============================================================
    \section{Soluciones Existentes}
    
    Al ser un tipo de sistema común la cantidad de sofware que ya existe es enorme:
    
    \begin{itemize}
        \item Tenemos por ejemplo \emph{SICAR}: Este es un gestor para diferentes giros con muchas aplicaciones, compra venta, pago de servicios, inventario, monedero electrónico.
    
        En este sistema tenemos que el precio de compra es de \$4,000.00 y además no cuenta con facturación ni gestionamientos entre empleados y jefes.
    
    
        \item Con \emph{ASPEL} y \emph{COMPAQ} tenemos un sistema de facturación, control de usuarios, uso touch... Son cuestiones por las que pagas mensualmente una cantidad, tampoco tiene gestionamientos entre empleados y jefes.
    
        \item El software \emph{Punto de Venta eleventa} ofrece muchas menos caracteristicas dejando básicamente facturación, ganancias y reduccion de costos. Eleventa tiene una interfaz amigable.
    \end{itemize}
    
    Podemos ver que un sistema así es difícil que se obtenga solo para lo que necesitas o que tenga lo que quieres: Lo que nos diferencia es la interfaz amigable y que los usuarios puedan tener funcionalidades específicas requeridas como el gestionamiento de permisos de usuarios.

    \section{Objetivos Generales}
    
        \begin{itemize}
            \item 
                Buscamos crear un sitema que permita organizar toda la administración de una tienda
                de abarrotes y todas las subdivisiones de la organización de manera concisa, confiable
                y segura.

            \item
                Queremos que dicha aplicación web pueda ser usada con gran facilidad, importando lo menos
                posible sistema operativo que usen nuestros clientes para usar la aplicación así como el
                disposivito que utilicen, sea un tablet, un celular o una computadora (de cualquier gama).

            \item
                Buscamos que la interfaz final que vean los usuario sea agradable a la vista, que sea intuitiva
                y siga las directices modernas de la creación del UX.
            
            \item
                Buscamos que sea de fácil mejoramiento, que dicha aplicación esté bien documentada
                y que su arquitectura sea capaz der ser actualizable y corregir de la manera más rápida
                cualquier error que tuviera nuestra aplicación.

            \item
                Buscamos que la información que tenga nuestra aplicación web en la base de datos sea de fácil
                acceso para la creación posterior de nuevas interfaces y aplicaciones, así como para su
                mantenimiento y escalabilidad.

            \item
                Buscamos la mayor seguridad en los datos, de tal manera que sean lo más confiables posible.

            \item
                Buscamos que nuestra aplicación web permita diferentes perfiles de usuarios, para mejorar la
                confiabilidad y seguridad del sistema permitiendo a cierto tipo de usuarios solo hacer ciertas
                actividades y tener la capacidad de leer / modificar ciertos datos dentro del sistema.

        \end{itemize}

        \subsection{Objetivos Particulares}
        
            \begin{itemize}
                \item 
                    Crea un conjunto de menus que sean una Single Web App, es decir, una aplicación web
                    que solo necesita ser cargada una sola vez y desde ahí trabajar con conexión con el
                    servidor mediante Sockets o JSON.
                
                \item
                    Crear una SWA que nos permita usar de INPUT una máquina lectora de código de barras,
                    necesitamos que podamos acceder a la base de datos de nuestros productos para crear
                    una especie de \Quote{carrito de compras} que nos diga cuánto debe pagar un cliente.
                    
                    Esta tiene que ser capaz de dar el mismo producto para diferenetes códigos de barras,
                    por ejemplo, para todos los Boings de medio litro, sin importar el sabor.
                    
                    Tiene que ser capaz de poder procesar ciertos prodcutos no solo por cantidad entera sino también
                    por fragmento, por ejemplo, 0.5 de Huevo o podemos hacerlo por precio, por ejemplo, 
                    \$ 20 pesos de longaniza.
                    
                    Además, esta Web App tiene que ser capaz de poder elegir entre diversos usuarios para
                    saber quién es el encargado actual.
                    
                \item
                    Ser capaz de mostrar los datos más relevantes de nuestra base de datos, es decir,
                    poder mostrar la cantidad que se ha vendido en los últimos dias, quién de los encargados
                    vende más, cuáles son los productos más vendidos.
                
            \end{itemize}
